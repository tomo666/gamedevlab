# JBScript


## JBScriptとは？

JBScript（ジェービースクリプト）は、**ゲームを作るためのプログラミング言語**です。
書き方はC#やPHPにかなり似ていて、ゲームの中の「こう動いてほしい」を書いて実行することができます。

たとえば…
- ボタンが押されたら、キャラが動く
- 会話ウィンドウを出す
- 画像を表示して、フェードさせる
- マップにキャラを置く

みたいな「ゲームあるある」を、短いコードで書けるように作られたプログラミング言語です。ここでは、このプログラミング言語のことを「スクリプト」と呼びます。

```csharp title="JBScriptのコードの例"
using JBScript;
using JBGE.GameEngine;

namespace MyGame {
  public static class Program {
    // はじめに1回だけ呼ばれる
    [JBScript_Start]
    public static void Start() {
      Common.SetScreenScale(5f);
    }

    // 毎フレーム呼ばれる（ゲームの心臓）
    [JBScript_Update]
    public static void Update() {
      // game loop
    }
  }
}
```

## 全体図をざっくり描くと

ゲームを動かす流れを、いちばん簡単に描くとこうです。

```
JBScript（台本）
   ↓
JBVM（台本を読む実行箱）
   ↓
JBGE（ゲームエンジンをまとめる土台）
   ↓
Unity / Godot / Unreal / RealityKit（舞台となる大きなゲームエンジン）
```

**台本（JBScript）を変えずに、舞台（エンジン）を変えられる**。
これがJBScript + JBGE + JBVMを使ったゲーム作りの仕組みです。

---

## まず、スクリプトって何？

ゲームには大きく2つの役割があります。

- **エンジン**: 画面に絵を出したり、音を鳴らしたり、当たり判定をしたりする「土台」
- **スクリプト**: 「いつ、何をするか」を決める「台本（ルール）」

JBScriptは、この**台本**の部分を書きやすくするための言語です。

---

## どうしてC#っぽいの？

理由はシンプルです。C#はUnityやGodotでもお馴染みの言語で、**読みやすくて、書きやすい**からです。

- `if` や `for` などが分かりやすい
- `class` や `static` など、整理して書ける
- C#経験者ならすぐ始められる

さらに、もし将来
- 「このスクリプト、もっと高速にしたい！」
- 「エンジン側（UnityやGodotなど）に移植したい！」

となったときも、最初からC#っぽい書き方をしておくと移植がラクになります。

たとえば、プロトタイプはJBScriptで素早く作って、
動きが固まったらUnity/Godot側にC#として移す…という流れが取りやすい、というメリットがあります。

---

## JBScriptでできること

JBScriptは、主にゲームエンジン「**Jelly Bean Game Engine（以下JBGE）**」に命令を送ってゲームを動かすために設計されています。例えば：

- UI（テキスト、画像、レイヤー）を表示する
- マップやアクター（キャラ）を表示する
- キー入力やゲームパッド入力を見る
- 画面にメッセージを出す

---

### Jelly Bean Game Engineとは？

JBGE（Jelly Bean Game Engine）は、JBScriptから呼び出して使う**ゲーム用エンジン**です。
JBGEが担当するのは、ゲームの「土台」になる部分です。例えば…

- 画面に絵を出す（UI、マップ、スプライトなど）
- 入力を読む（キーボード、ゲームパッドなど）
- 当たり判定や移動を処理する
- 画像やフォントなどの素材を読み込む

つまり、**ゲームのルールはJBScript、表示や入力などの土台はJBGE**が担当します。

### JBGEは「ほかのゲームエンジンの上」で動く

JBGEは数多くあるゲームエンジンのひとつですが、ひとつ大きな特徴があります。
それは、JBGEが**Unity / Godot / Unreal みたいな有名ゲームエンジンの上で動く**ということです。

イメージとしては：

- Unity / Godot / Unreal = 大きな「遊園地」
- JBGE = その中に作る「アトラクション用の舞台」

つまり、JBGEはゼロから全部を作るエンジンというより、**既存の大きなエンジンの力を借りて、同じルールでゲームを動かすための共通レイヤー**です。

この仕組みを実現するために、JBGEは **Jelly Bean Virtual Machine（JBVM）** を用意して、JBScriptとゲームエンジンの間を橋渡ししています。

---

### Jelly Bean Virtual Machine（JBVM）とは？

JBVM（Jelly Bean Virtual Machine）は、JBScriptを動かすための**小さな実行箱**です。

難しく聞こえるけど、イメージはこんな感じです。

- **本物のゲーム機**: 決まったルールで動く
- **JBScriptの実行環境（VM）**: “小さなゲーム機っぽい箱”の中で、同じルールで動く

JBScriptはそのままだとWindowsやMac、iOSやAndroidなど、各OSや環境で動かし方がバラバラになります。そこで、JBVMという共通ルールの箱の中で動かすことで、同じスクリプトをいろんな環境に持っていきやすくしているのです。

つまり：
- **JBScript = 台本**
- **JBVM = 台本を読む人（読み方のルールが決まっている）**
- **JBGE = 舞台装置**

という役割分担です。

何でJBVMのような仮想マシンがゲーム作りに必要なの？
- どのPC（やゲーム機）でも、同じように動かしやすい
- 実行の仕方が決まっているので、バグが追いやすい
- 一度コードを書いたら、環境ごとにコードを修正しなくても実行できる


### JBVMがあるから「一度スクリプトを書けばどこでも走る」

- Unity版JBVM
- Godot版JBVM
- Unreal版JBVM
- RealityKit版JBVM

…のように、JBVMは各ゲームエンジン上で動くように設計されているので、
**JBScriptで一度コードを書くだけで、同じゲームがいろんなエンジンで動かせます。**

:::note
もちろん、エンジンごとに得意・不得意はあります（3Dが強い、軽い、ツールが充実…など）。
でもJBScriptの「台本」は共通なので、将来的にゲームエンジンを変更した場合でも、ゲームのルールを作り直す手間が減ります。より高度なゲーム開発がしたくなったら、Unityなどのゲームエンジンをそのまま使いつつ、JBScript+JBVM+JBGEも同時に使う、といったハイブリッドな開発もできるようになっています。
※このドキュメントでは、最初はVMの細かい話をしません。必要になったら、あとで少しずつ説明します。
:::

## JBScriptを使うメリット

- **書き方がC#っぽい**ので、読みやすい・学びやすい
- **ゲーム向けの命令**が最初から揃っている
- **どこでも同じように動く**ように設計できる
- エンジンと台本が分かれるので、ゲームの作りが整理しやすい

---

## JBScriptを使うデメリット

- ふつうのC#そのままではない（使える機能に制限がある）
- エンジンに用意されていない機能は、スクリプトだけでは実現できない
- メモリ管理（`Destroy()`など）を意識する場面がある
